================================================================================
PROJECT OVERVIEW
================================================================================

The Semantic Compliance Review Agent is an AI-assisted compliance review system
built for the Google x Kaggle AI Agents: Intensive Vibe Coding Capstone Project.

It is NOT an autonomous code modification tool.

The purpose of the agent is to identify potentially risky human-written text
inside source code repositories and assist developers in producing safer,
more professional, and more compliant codebases.

The project focuses on comments, documentation, TODO/FIXME notes, docstrings,
and selected human-readable strings that traditional static analysis tools
typically ignore.

Core principle:

    "Assist the developer.
     Never replace the developer."

================================================================================
THE PROBLEM
================================================================================

Developers frequently leave behind information inside source code that can
create security, legal, intellectual property, compliance, or reputation risks.

Examples:

- Internal project codenames
- Security shortcuts
- Temporary credentials
- Unprofessional comments
- Sarcastic remarks
- TODO notes containing sensitive information
- Internal system descriptions
- Risky documentation

Traditional linters and static analysis tools generally cannot understand
the semantic meaning behind this text.

Humans can.

Modern LLM-powered agents can assist.

- Internal Codename Exposure
- Intellectual Property Risk

SPECIAL NOTE: INTERNAL CODENAME EXPOSURE

Internal codename detection presents a unique challenge because codenames are
often organization-specific and cannot always be identified through semantic
analysis alone.

To improve reliability, the system combines:

- Organization-defined sensitive term lists
- Project-specific context
- Semantic analysis

Known sensitive terms can be detected deterministically.

Unknown suspicious references may be flagged through semantic review and
assigned an appropriate confidence level.

PHASE 4.5 — AGENT REVIEW DESIGN

- Document the `ReviewableText[] + ReviewContext -> Agent Review -> Finding[]` flow
- Document the `Finding` contract
- Lock the review boundary before Phase 5 coding begins

Approved review boundary:

- `agent_review.review(reviewable_texts, review_context) -> list[Finding]`

Approved `Finding` fields:

- `id`
- `reviewable_text_id`
- `category`
- `severity`
- `confidence`
- `detection_method`
- `source_text`
- `line_start`
- `line_end`
- `explanation`
- `recommendation`
- `suggested_replacement`

Approved finding categories:

- `SECURITY_RISK`
- `PROFESSIONALISM_RISK`
- `COMPLIANCE_RISK`
- `INTERNAL_CODENAME_EXPOSURE`
- `INTELLECTUAL_PROPERTY_RISK`
- `REPUTATION_RISK`

Approved severity values:

- `LOW`
- `MEDIUM`
- `HIGH`
- `CRITICAL`

Approved confidence values:

- `LOW`
- `MEDIUM`
- `HIGH`

Approved detection methods:

- `TERM_MATCH`
- `SEMANTIC_ANALYSIS`
- `HYBRID`

Detection method rules:

- `TERM_MATCH` = configured sensitive term detected, no additional semantic
  reasoning needed
- `SEMANTIC_ANALYSIS` = meaning, tone, or implication based finding with no
  configured sensitive-term match
- `HYBRID` = configured sensitive term detected plus additional semantic risk
- do not use `HYBRID` just because semantic reasoning was used
- do not use `HYBRID` just because the finding is high severity
- do not use `HYBRID` just because a suggested replacement exists

Important MVP rules:

- `TERM_MATCH` findings are HIGH confidence by construction
- zero findings is a successful review result
- retry once if structured output cannot be parsed
- `suggested_replacement` stays optional
- use `suggested_replacement` when a HIGH-confidence finding has an obvious,
  safe, neutral rewrite based only on the source text; otherwise use `null`
- suggested replacements must preserve intent while removing risky, sensitive,
  or unprofessional wording
- suggested replacements must not introduce secrets, internal codenames,
  credentials, or speculative information
