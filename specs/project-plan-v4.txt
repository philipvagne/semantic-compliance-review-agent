================================================================================
PROJECT OVERVIEW
================================================================================

The Semantic Compliance Review Agent is an AI-assisted compliance review system
built for the Google x Kaggle AI Agents: Intensive Vibe Coding Capstone Project.

It is NOT an autonomous code modification tool.

The purpose of the agent is to identify potentially risky human-written text
inside source code repositories and assist developers in producing safer,
more professional, and more compliant codebases.

The project focuses on comments, documentation, TODO/FIXME notes, docstrings,
and selected human-readable strings that traditional static analysis tools
typically ignore.

Core principle:

    "Assist the developer.
     Never replace the developer."

================================================================================
THE PROBLEM
================================================================================

Developers frequently leave behind information inside source code that can
create security, legal, intellectual property, compliance, or reputation risks.

Examples:

- Internal project codenames
- Security shortcuts
- Temporary credentials
- Unprofessional comments
- Sarcastic remarks
- TODO notes containing sensitive information
- Internal system descriptions
- Risky documentation

Traditional linters and static analysis tools generally cannot understand
the semantic meaning behind this text.

Humans can.

Modern LLM-powered agents can assist.

CONFIDENCE MODEL
================================================================================

HIGH CONFIDENCE

Examples:
- Credential exposure
- Configured sensitive term match
- Obvious professionalism violations

Action:
- Suggest remediation

MEDIUM CONFIDENCE

Examples:
- Potentially sensitive references
- Ambiguous language

Action:
- Review recommended

LOW CONFIDENCE

Examples:
- Weak signal
- Uncertain interpretation

Action:
- Informational only

Example:

- Sensitive business domains

This context improves detection reliability while reducing false positives.

RISK CATEGORIES
================================================================================

- Security Risk
- Professionalism Risk
- Compliance Risk
- Internal Codename Exposure
- Intellectual Property Risk
- Reputation Risk

SPECIAL NOTE: INTERNAL CODENAME EXPOSURE

Internal codename detection presents a unique challenge because codenames are
often organization-specific and cannot always be identified through semantic
analysis alone.

To improve reliability, the system combines:

- Organization-defined sensitive term lists
- Project-specific context
- Semantic analysis

Known sensitive terms can be detected deterministically.

Unknown suspicious references may be flagged through semantic review and
assigned an appropriate confidence level.

SECURITY PRINCIPLES
================================================================================

- Preserve code structure
- Preserve business logic
- Maintain an audit trail
- Explain all findings
- Prefer reporting over modification
- Keep API keys and secrets out of the repository

================================================================================
GUARDRAILS
================================================================================

The agent must NEVER:

- Remove functions
- Remove classes
- Remove imports
- Remove variables
- Modify executable code
- Change application behavior
- Commit changes automatically
- Delete content automatically

Version 1:

Comments and documentation may receive suggested replacements.
Human-readable strings are report-only.
